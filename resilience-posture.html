CIB Unified Ingress: Resilience Posture

Classification: Internal
Status: Draft for Review
Owner: Head of Architecture
The platform is designed so that every failure mode below the CDN vendor level is handled automatically with no user impact. Pod failures, regional failures, session continuity, and revocation propagation all recover without human intervention. CDN vendor failure is the sole case requiring manual action — a deliberate trade-off accepted because vendor-level outages are rare, short-lived, and the operational cost of active/active across two CDN providers outweighs the marginal availability gain. An ultimate DR path exists to bypass CDN layers entirely and route directly to the perimeter in a catastrophic scenario.

JPM-operated authoritative NS. Full organisational control, on-premises DNS infrastructure.

Cloudflare Nameservers
Cloud-hosted secondary NS. Serves the same zone data. Active in parallel — not a standby.
 Instant
Both NS sets are active simultaneously. Resolvers query whichever responds first. Failure of one set is transparent — the other continues serving all DNS queries without any failover action required.
NS records for CIB ingress domains delegate to both self-hosted and Cloudflare nameservers
Resolvers receive both NS sets and will use whichever responds — no single point of failure at the DNS layer
Zone data must remain in sync across both sets at all times
GTM records (short TTL, health-check driven) served through whichever NS set the resolver contacts
Splitting nameservers across self-hosted and Cloudflare eliminates the DNS layer as a single point of failure. A full DNS outage would require both the JPM on-premises DNS infrastructure and Cloudflare's global DNS network to fail simultaneously — an extremely unlikely scenario.

This is distinct from the L2 CDN/WAF layer where Cloudflare is a warm standby. At the DNS layer, Cloudflare nameservers are active all the time. Both NS sets serve live traffic. The split is about resilience, not cost or routing preference.

Note Zone synchronisation between self-hosted and Cloudflare nameservers is a critical operational dependency. A lag in propagating GTM record updates or DR DNS changes across both NS sets could cause inconsistent resolution during an incident. Sync latency should be monitored and bounded.

GTM is the automatic regional failover control point. The composite health check must define a meaningful region-unhealthy threshold — not a simple ping — to avoid false failovers. Hysteresis configuration is critical: a region that recovers and immediately takes traffic back risks flapping.

Note Regional health check design must be agreed before GTM configuration begins. This is a functional decision, not a tuning exercise.

Active/active between Akamai and Cloudflare is not viable. WAF rule consistency is the critical problem: splitting traffic across two CDN/WAF vendors requires maintaining identical OWASP rulesets, custom CIB rules, bot management policies, and TLS configuration on both platforms simultaneously. In practice these drift. The security posture becomes the weaker of the two at all times. For a regulated financial institution this is not acceptable.

Vendor-level CDN outages are rare and brief — the June 2021 Akamai outage lasted approximately one hour. Manual recovery within that window is operationally acceptable.

Note Cloudflare config must be exercised quarterly via DR test. An untested DR path is not a DR path.

Sidecar isolation is the key design decision at L4. Running auth components as sidecars rather than shared services means a failing OPA instance or Session Validator affects only its own pod. The rest of the gateway fleet continues unaffected. This is fundamentally more resilient than a shared auth service with a single failure domain.

This is the layer that makes regional failover invisible to users. Without active/active session state, a regional failover forces re-authentication for every active user — a significant and visible disruption in a trading context. The Kafka replication model ensures session state is already present in the target region before failover occurs.

Ultimate DR Direct Perimeter Route — CDN Bypass
What it is
A last-resort path that routes traffic directly to PSaaS+ (GKP path) or the AWS WAF layer, bypassing both Akamai and Cloudflare entirely. Used only when both CDN/WAF vendors are simultaneously unavailable, or when a security incident requires emergency removal of all CDN intermediaries.

This path exists because the perimeter (PSaaS+ and AWS WAF) provides a base level of protection and routing capability independent of any CDN vendor. It is not a normal operating mode.
What is lost
  • CDN caching and acceleration
  • Edge TLS termination at Akamai/Cloudflare PoPs
  • Akamai Kona WAF and Cloudflare WAF protection
  • DDoS mitigation at CDN edge
  • Bot management and credential stuffing protection
  • GTM GeoDNS steering (manual DNS required)

The platform remains functional for authenticated users but is exposed to a broader threat surface and higher origin load.
Activation and recovery objective
Activation requires a manual DNS change to point CIB hostnames directly at PSaaS+/AWS WAF endpoints, coordinated security sign-off, and agreement that the risk of operating without CDN WAF protection is acceptable for the outage duration.

RTO: hours. This is not a fast-activation path. It exists to ensure continuity is possible in a catastrophic scenario — not to provide rapid recovery.

Runbook, pre-approved DNS change records, and security sign-off process must be documented and tested before go-live.
